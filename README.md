# Velox Trade

Полнофункциональный маркетплейс игровых предметов: Next.js 14 + TypeScript + TailwindCSS + Firebase + EmailJS.

Всё, что описано ниже, реально работает (не демо/заглушки) при условии, что Firebase настроен по инструкции.

**Что исправлено в этой версии для чистого деплоя на Vercel:**
- Убран риск зависания сборки из-за отсутствующего ESLint-конфига (`next.config.js` теперь явно отключает линт на билде)
- Исправлена ошибка `useSearchParams() should be wrapped in a suspense boundary`, которая ломала сборку на странице каталога
- Убраны неиспользуемые зависимости, ускоряющие установку
- Оба админ-аккаунта уже прописаны в `.env.local.example` и `firestore.rules`

**Что нового в этой версии:**
- Сайт переименован в **Velox Trade**
- Загрузка изображений теперь через файлы (Firebase Storage), а не по ссылке — товары, игры, реклама, аватары (см. раздел про Storage ниже)
- Полное меню Telegram-бота с кнопками: Обратная связь → Пополнение/вывод и Поддержка, а также Сотрудничество — все сообщения от пользователей уходят администратору напрямую в Telegram
- Рассылка в Telegram теперь поддерживает фото + кнопку со ссылкой
- Юзернеймы, публичные профили продавцов, отзывы, чат по заказу, жалобы на сделки — раздел «Маркетплейс: продавцы, отзывы, споры» ниже
- Исправлен краш каталога/админки при невалидной ссылке на картинку (баг с `Failed to parse src "а"`)
- Мобильная навигация в админ-панели (раньше боковое меню пропадало на телефоне без замены)

## ⚡ Частая ошибка: "Missing or insufficient permissions"

Если при регистрации/входе видишь `FirebaseError: Missing or insufficient permissions` — значит правила Firestore (`firestore.rules`) ещё не опубликованы в консоли Firebase, и сейчас действуют правила по умолчанию ("запретить всё").

1. Зайди на https://console.firebase.google.com/project/blade-shoop/firestore/rules (именно `/firestore/rules`, не `/database/.../rtdb/` — см. раздел 3 про Firestore vs Realtime Database)
2. Выдели всё в редакторе (Ctrl+A), удали, вставь содержимое файла `firestore.rules` из этого архива
3. Нажми **Publish**, подожди 1-2 минуты, обнови страницу сайта

## 🔐 Настройка входа по коду в Telegram

Это отдельная, более сложная функция — она использует Firebase Admin SDK и Telegram Bot API напрямую с сервера (через API-роуты Next.js), а не только клиентский Firebase, как остальной сайт. Без настройки ниже кнопки привязки/входа будут выдавать ошибку — это ожидаемо, пока не выполнены шаги.

**Важное ограничение:** приём сообщений от Telegram-бота (вебхук) требует публичного HTTPS-адреса. На `localhost` это физически невозможно — Telegram не может достучаться до твоего компьютера. Значит, привязка Telegram и вход по коду заработают **только после деплоя на Vercel**, не в `npm run dev`. Локально можно проверить только то, что кнопки не выдают ошибок конфигурации.

### Шаг 1 — токен Telegram-бота

1. В Telegram напиши **@BotFather**
2. Если бот `@veloxtrade_robot` уже существует и это твой бот — напиши `/mybots` → выбери его → **API Token**
3. Если бота ещё нет — `/newbot`, следуй инструкциям, в конце получишь токен вида `123456789:AAH...`
4. Впиши токен в `.env.local` и позже в Vercel:
   ```
   TELEGRAM_BOT_TOKEN=твой_токен
   ```

### Шаг 2 — сервисный ключ Firebase Admin

1. Firebase Console → шестерёнка рядом с «Project Overview» → **Настройки проекта**
2. Вкладка **Service accounts** → **Generate new private key** → скачается `.json`-файл
3. Открой этот файл, оттуда возьми три значения:
   - `project_id` → `FIREBASE_ADMIN_PROJECT_ID`
   - `client_email` → `FIREBASE_ADMIN_CLIENT_EMAIL`
   - `private_key` → `FIREBASE_ADMIN_PRIVATE_KEY` (это длинная строка с `\n` внутри — скопируй её целиком в кавычках как есть, ничего не меняя)
4. Впиши все три в `.env.local`

**Держи этот файл в секрете** — он даёт полный доступ к твоей базе Firebase в обход всех правил безопасности. Никогда не клади его в GitHub-репозиторий (в `.gitignore` уже добавлено правило на `.env.local`, но проверь, что сам JSON-файл тоже не попадёт в git, если сохранишь его в папку проекта).

### Шаг 3 — задеплой на Vercel с этими переменными

Добавь все переменные из шагов 1-2 (`TELEGRAM_BOT_TOKEN`, `TELEGRAM_WEBHOOK_SECRET`, `FIREBASE_ADMIN_PROJECT_ID`, `FIREBASE_ADMIN_CLIENT_EMAIL`, `FIREBASE_ADMIN_PRIVATE_KEY`) в Vercel → Settings → Environment Variables, вместе с остальными (см. раздел «Деплой» ниже). Задеплой.

### Шаг 4 — подключить вебхук (один раз, после деплоя)

Открой в браузере (замени `<ТОКЕН>` и `<ДОМЕН>` на свои, `<СЕКРЕТ>` — на то же значение, что в `TELEGRAM_WEBHOOK_SECRET`, либо убери `&secret_token=...` вообще, если не задавал этот секрет):

```
https://api.telegram.org/bot<ТОКЕН>/setWebhook?url=https://<ДОМЕН>.vercel.app/api/telegram/webhook&secret_token=<СЕКРЕТ>
```

Должен прийти ответ `{"ok":true,"result":true,...}`. Это всё — теперь бот будет присылать все сообщения на твой сайт.

### Проверка

1. Зайди на сайт → Профиль → Безопасность → «Привязать Telegram» → «Открыть Telegram-бота» → нажми **Start** в самом Telegram
2. Вернись на сайт, нажми «Обновить статус» — должно показать «Привязан»
3. Выйди из аккаунта, зайди на `/auth/login` → вкладка «Код в Telegram» → введи email → должен прийти код в бота → введи его → вход должен пройти

Если что-то не работает — проверь в Vercel → Deployments → последний деплой → **Functions** → логи `api/telegram/webhook` и `api/auth/request-code`, там будут точные ошибки.

## 🤖 Меню Telegram-бота (Обратная связь / Сотрудничество)

Помимо входа по коду, бот теперь работает как полноценная точка приёма обращений. Как только пользователь пишет `/start` без кода привязки, бот показывает меню:

- **💬 Обратная связь** → подменю **Пополнение/вывод** и **Поддержка** — что бы пользователь ни написал после выбора пункта, сообщение целиком уходит администратору в Telegram
- **🤝 Сотрудничество** → инструкция написать «хочу сотрудничать с Velox Trade» — тоже пересылается администратору
- **📢 Наш канал** — кнопка появляется, только если заполнена `NEXT_PUBLIC_TELEGRAM_CHANNEL_URL` в `.env.local`

### Настройка

1. Токен бота уже вписан в `.env.local.example` (`TELEGRAM_BOT_TOKEN`). Если нужно сменить — @BotFather → `/mybots` → выбрать бота → API Token.
2. Впиши **личный Telegram chat_id администратора** в `TELEGRAM_ADMIN_CHAT_ID` — именно на этот chat_id бот шлёт все сообщения от пользователей (обратная связь, сотрудничество) и уведомления о жалобах на сделки. Это числовой ID, не username. Он уже проставлен по умолчанию (`7040863301`) — если это не твой ID, узнать свой можно, написав своему же боту `/start` и посмотрев апдейт в `https://api.telegram.org/bot<ТОКЕН>/getUpdates`, поле `message.from.id`.
3. (Необязательно) `NEXT_PUBLIC_TELEGRAM_CHANNEL_URL=https://t.me/твой_канал` — добавит кнопку подписки на канал в главном меню бота.
4. Не забудь заново подключить вебхук после смены токена (см. Шаг 4 раздела выше про вход по коду) — вебхук привязан к конкретному токену.

## 🖼️ Загрузка изображений файлами (Vercel Blob)

Товары, игры, реклама и аватары загружаются как файлы (JPG/PNG/WEBP/GIF/SVG, до 5 МБ), а не вставляются по ссылке. Хранилище — **Vercel Blob**, а не Firebase Storage: это осознанный выбор, чтобы не требовать платный тариф Blaze в Firebase — у Vercel Blob свой бесплатный лимит, доступный сразу после регистрации, без привязки карты.

### Настройка (только на самом Vercel — не в Firebase Console)

1. Зайди в свой проект на **vercel.com** → вкладка **Storage** → **Create Database** → выбери **Blob**
2. Дай хранилищу имя (можно любое) → **Create**
3. Vercel сам подключит переменную `BLOB_READ_WRITE_TOKEN` к проекту — никаких файлов правил создавать не нужно, в отличие от Firestore/Storage у Firebase
4. Если после этого сайт на проде уже был задеплоен — сделай **Redeploy** (Vercel → Deployments → ⋯ → Redeploy), чтобы новая переменная подхватилась

### Для запуска на localhost

Переменная `BLOB_READ_WRITE_TOKEN` в `.env.local` пустая по умолчанию — на Vercel она подставляется автоматически, а локально нужно скопировать её вручную:
- Vercel → твой проект → Storage → выбери Blob-хранилище → там будет команда для копирования переменной, либо
- Если установлен Vercel CLI: `vercel env pull .env.local` подтянет все переменные разом

### Права на загрузку

Сервер сам проверяет, кто загружает файл (по Firebase ID-токену, который передаётся с каждой загрузкой):
- Товары, игры, реклама — только админ
- Аватар — любой залогиненный пользователь

Если увидишь ошибку **"Загружать сюда может только администратор"** или **"Сессия истекла"** — это она и есть, разбирается по смыслу сообщения. Любая другая ошибка — подробности в консоли браузера (F12 → Console).

## 🛒 Маркетплейс: продавцы, отзывы, споры

- **Юзернейм и аватар** (`/profile`): ник и аватар можно менять раз в 7 дней, юзернейм — без ограничений (используется в ссылке `/seller/username`)
- **Публичный профиль продавца** (`/seller/[username]`): аватар, бейджи, рейтинг, отзывы, товары этого продавца (поле `sellerId` у товара — админ указывает его при создании/редактировании товара, по умолчанию `store`, если товар не привязан к конкретному пользователю)
- **Подтверждение сделки**: после оплаты заказ висит в статусе «Ждёт подтверждения» → покупатель либо подтверждает получение (открывает возможность оставить отзыв), либо открывает жалобу
- **Жалобы на сделки** (`/admin/disputes`): администратору сразу приходит уведомление в Telegram (на `TELEGRAM_ADMIN_CHAT_ID`), одобрение/отклонение — вручную в админке. Возврат средств при одобрении делается вручную через Пользователи → изменить баланс — автоматического возврата пока нет
- **Чат по заказу**: покупатель и продавец общаются в контексте конкретного заказа, администратор видит и может писать в любой чат

**Важная честная оговорка:** баланс списывается сразу при оформлении заказа (как и было с самого начала проекта) — это не настоящий эскроу с заморозкой денег до подтверждения сделки, а обычное списание. «Подтверждение сделки» здесь работает как условие для отзыва и как сигнал администратору, но не как техническая заморозка средств. Полноценный эскроу — это отдельная, более крупная доработка (нужен отдельный «замороженный» баланс и логика его перевода продавцу).

## Установка

### 1. Зависимости

```powershell
npm install
```

### 2. Файл окружения

```powershell
Copy-Item .env.local.example .env.local
Get-Content .env.local
```

Все нужные ключи (Firebase, EmailJS) уже вписаны в `.env.local.example`.

### 3. Включить Firestore в Firebase

⚠️ Не перепутай **Firestore Database** (нужна проекту) с **Realtime Database** (не используется, синтаксис правил другой — вставка правил Firestore туда даёт `Parse error`).

1. https://console.firebase.google.com/project/blade-shoop/firestore
2. Если нет базы — **Create database** → **Start in production mode** → любой регион → **Enable**
3. Вкладка **Rules** (адрес должен содержать `/firestore/rules`, а не `/database/.../rtdb/`)
4. Вставь содержимое `firestore.rules` из архива, **Publish**

### 4. Админы уже настроены

В этой версии в `.env.local.example` и `firestore.rules` уже прописаны два админ-аккаунта (UID `s4Cbs8BFRwd9CrPYNkQBQzbVlQG3` и `DAdWBqr7i8UWc1nT0NVVMuJ1DJQ2`). Если это твои реальные UID из Firebase Authentication — ничего менять не нужно, просто:

1. Убедись, что оба аккаунта зарегистрированы на сайте (через обычную форму регистрации)
2. Опубликуй `firestore.rules` в консоли Firebase (см. раздел выше)
3. Готово — у обоих в личном кабинете появится «Админ-панель»

**Если нужно поменять/добавить админа:**
1. Firebase Console → **Authentication** → **Users** → скопируй нужный UID
2. В `.env.local` пропиши в `NEXT_PUBLIC_ADMIN_UIDS` через запятую **без пробелов**
3. В `firestore.rules` найди функцию `isAdmin()` и замени/добавь UID в списке (можно как угодно — один, два, три и больше строк через запятую)
4. Сохрани, снова опубликуй правила в консоли Firebase
5. Перезапусти `npm run dev`

### 5. (Опционально) Категории игр одним кликом

Каталог по умолчанию пустой. Если не хочешь вводить игры вручную:

```powershell
npm run seed
```

Создаст только категории игр (без товаров) из `src/lib/mockData.ts`. Если правила Firestore ещё не разрешают анонимную запись — временно поставь `allow read, write: if true;` (см. блок ниже), засей, и сразу верни `firestore.rules` обратно.

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /{document=**} {
      allow read, write: if true;
    }
  }
}
```

### 6. Запуск

```powershell
npm run dev
```

## 📧 Если письма не приходят

В проекте два независимых источника писем — если письма не доходят, важно понять, какой из них не работает.

### A) Письма из Firebase (подтверждение email, восстановление пароля)

Эти письма отправляет сам Firebase Authentication, без EmailJS.

1. Проверь папку **«Спам»** — письма от `noreply@blade-shoop.firebaseapp.com` часто туда попадают, особенно первое время
2. Firebase Console → **Authentication** → **Sign-in method** — убедись, что провайдер **Email/Password** включён (Enabled)
3. Firebase Console → **Authentication** → **Templates** — убедись, что шаблоны «Email address verification» и «Password reset» включены
4. Учитывай: Firebase ограничивает частую повторную отправку (не чаще раза в минуту) — если только что нажал «Отправить письмо» второй раз подряд, получишь ошибку `auth/too-many-requests`, это не баг
5. На бесплатном тарифе Spark есть дневной лимит писем — для разработки/теста его почти невозможно исчерпать, но если тестируешь массово, имей в виду

### B) Письма через EmailJS (рассылка из админки)

1. Зайди в `/admin` → **Реклама и рассылки** → вкладка **Рассылка** → нажми кнопку **«Тест»** — она пришлёт письмо на твой собственный email и покажет точную причину ошибки, если она есть
2. Если тест не проходит — открой консоль браузера (F12 → Console) и посмотри код ошибки от EmailJS:
   - **403 / 401** — почти всегда проблема с доменом. Зайди в личный кабинет EmailJS → **Account** → **Security** → **Allowed origins/domains** и добавь `http://localhost:3000` (для разработки) и домен, куда задеплоишь сайт на Vercel
   - **400** — не совпадают названия переменных в шаблоне EmailJS. Зайди в EmailJS → **Email Templates** → `template_qo1n6m8` и проверь, что в теле шаблона используются переменные `{{to_email}}`, `{{to_name}}`, `{{subject}}`, `{{message}}`, `{{button_text}}`, `{{button_link}}` — именно эти названия использует код в `src/lib/emailjs.ts`. Если в твоём шаблоне переменные называются иначе (например, `{{email}}` вместо `{{to_email}}`) — either переименуй переменные в шаблоне EmailJS, либо поправь названия полей в `sendEmail(...)` в `src/lib/emailjs.ts`, чтобы они совпадали
3. Проверь, что сервис почты в EmailJS (Gmail/Outlook/SMTP) подключён и не просит повторной авторизации — это видно в EmailJS → **Email Services**
4. Бесплатный тариф EmailJS ограничен ~200 письмами в месяц — для рассылки по большой базе пользователей этого может не хватать, тогда стоит перейти на платный тариф EmailJS либо перенести рассылку на серверный сервис (Resend, SendGrid) через Cloud Function

## 🚀 Деплой: GitHub → Vercel

### Шаг 1 — залить проект на GitHub

Если у тебя ещё нет Git-репозитория для проекта:

```powershell
cd путь\до\папки\blade-shop
git init
git add .
git commit -m "Первая версия Velox Trade"
```

Дальше на https://github.com:
1. Нажми **New repository**, дай название (например `blade-shop`), оставь **Private** (рекомендуется — в коде нет секретов, но лучше перестраховаться), не добавляй README/gitignore (они уже есть в проекте)
2. Нажми **Create repository**
3. GitHub покажет команды — выполни их у себя в PowerShell (замени `твой-логин`):
   ```powershell
   git remote add origin https://github.com/твой-логин/blade-shop.git
   git branch -M main
   git push -u origin main
   ```
4. Если попросит войти — используй свой GitHub-логин и **Personal Access Token** вместо пароля (GitHub больше не принимает обычный пароль для push; токен создаётся в GitHub → Settings → Developer settings → Personal access tokens)

### Шаг 2 — импортировать в Vercel

1. Зайди на https://vercel.com, войди через GitHub-аккаунт
2. **Add New...** → **Project**
3. Найди репозиторий `blade-shop` в списке, нажми **Import**
4. Vercel сам определит, что это Next.js — ничего в настройках сборки менять не нужно
5. Разверни **Environment Variables** и добавь **все** переменные из твоего `.env.local` — по одной, имя и значение:
   ```
   NEXT_PUBLIC_FIREBASE_API_KEY
   NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN
   NEXT_PUBLIC_FIREBASE_PROJECT_ID
   NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET
   NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID
   NEXT_PUBLIC_FIREBASE_APP_ID
   NEXT_PUBLIC_FIREBASE_MEASUREMENT_ID
   NEXT_PUBLIC_EMAILJS_SERVICE_ID
   NEXT_PUBLIC_EMAILJS_TEMPLATE_ID
   NEXT_PUBLIC_EMAILJS_PUBLIC_KEY
   NEXT_PUBLIC_TELEGRAM_BOT
   NEXT_PUBLIC_ADMIN_UIDS
   NEXT_PUBLIC_TELEGRAM_CHANNEL_URL
   TELEGRAM_BOT_TOKEN
   TELEGRAM_WEBHOOK_SECRET
   TELEGRAM_ADMIN_CHAT_ID
   FIREBASE_ADMIN_PROJECT_ID
   FIREBASE_ADMIN_CLIENT_EMAIL
   FIREBASE_ADMIN_PRIVATE_KEY
   ```
6. Нажми **Deploy** и подожди 1-2 минуты

**`BLOB_READ_WRITE_TOKEN` добавлять вручную не нужно** — она появится сама, как только подключишь Blob-хранилище (раздел «🖼️ Загрузка изображений файлами» ниже), Vercel привяжет её к проекту автоматически.

**Про `FIREBASE_ADMIN_PRIVATE_KEY`:** в скачанном JSON-файле это значение выглядит как одна строка с символами `\n` внутри (не настоящими переносами строк). Вставляй его в Vercel и в `.env.local` именно в таком виде — с буквально написанными `\n`, а не с реальными переносами строк. Код сам превращает `\n` обратно в переносы при чтении.

### Шаг 3 — донастройка после первого деплоя (обязательно)

Без этих трёх шагов сайт откроется, но вход не будет работать:

1. **Firebase Authorized domains.** Скопируй домен, который выдал Vercel (например `blade-shop-xyz.vercel.app`). Зайди в Firebase Console → **Authentication** → **Settings** → **Authorized domains** → **Add domain** → вставь этот домен. Без этого шага вход через Google на проде не сработает.
2. **EmailJS Allowed origins.** Зайди в EmailJS → Account → Security → добавь тот же домен Vercel в разрешённые. Без этого рассылка из админки будет давать 403 на проде, даже если локально работала.
3. **Firestore Rules с реальными UID.** Если ты вписывал `REPLACE_WITH_ADMIN_UID_1` — убедись, что в консоли Firebase (не только в файле в архиве) опубликованы правила с твоим настоящим UID, иначе `/admin` на проде будет недоступен.

После этого открой свой `https://....vercel.app` — сайт должен работать полностью так же, как на localhost.

### Обновление сайта после изменений в коде

```powershell
git add .
git commit -m "Описание изменений"
git push
```

Vercel автоматически пересоберёт и задеплоит новую версию за 1-2 минуты — никаких дополнительных действий не нужно.

## ⚠️ Важно про схему оплаты

Пополнение/вывод баланса работает через ручную заявку: пользователь создаёт заявку → пишет администратору в Telegram-бота → админ присылает реквизиты → подтверждает заявку в админке, баланс меняется автоматически.

Это рабочая, но полностью ручная схема без независимого подтверждения платежа — риск споров «перевёл деньги, баланс не обновили», путаницы из-за бот-двойников (в коде уже есть предупреждение пользователю об этом) и сложностей с возвратами. Если аудитория — преимущественно подростки (Roblox), стоит либо усилить прозрачность схемы, либо со временем подключить настоящий платёжный шлюз (ЮKassa, CloudPayments, CryptoCloud), который подтверждает оплату автоматически.
